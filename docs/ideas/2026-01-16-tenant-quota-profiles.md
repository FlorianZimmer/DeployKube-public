# Idea: Tenant Quota Profiles (Resources + Rate Limits)

Date: 2026-01-16  
Status: Draft

## Problem statement

DeployKube already enforces a baseline tenant posture when a namespace is labeled `darksite.cloud/rbac-profile=tenant`. This includes default-deny networking and a baseline `ResourceQuota`/`LimitRange` generated by Kyverno.

Today, quota/limit values are a **single cluster-wide default** baked into `tenant-baseline-generate`. This becomes a problem once we onboard real tenants:
- Different orgs/projects need different budgets (CPU/memory/storage/pods/object counts).
- We need an operator-friendly, GitOps-native way to express and review these budgets **per tenant/project** without editing shared policy templates for every onboarding.
- “Quota” must cover more than CPU/memory: storage, API object counts (control-plane protection), and traffic budgets (rate limiting / concurrency) so one tenant cannot starve shared infrastructure.

Goal: make tenant quotas a first-class, easy-to-change part of the tenant intent surface, while keeping the baseline drift-resistant and consistent across environments.

## Why now / drivers

- Multitenancy is already label-driven and GitOps-owned (`platform/gitops/tenants/**`), so adding quota intent fits the model.
- Tier S explicitly assumes “noisy/hostile tenant” scenarios; quotas are a prerequisite before scaling tenant onboarding.
- The current baseline quotas are intentionally conservative defaults; without a structured way to raise/lower budgets, operators will fork policies or hand-edit objects (drift risk).

## Proposed approach (high-level)

### 1) Introduce quota “profiles” selectable by label

Use a behavior label on tenant namespaces:
- `darksite.cloud/quota-profile=<small|medium|large|...>` — selects a preset containing:
  - `ResourceQuota/tenant-quota` (namespace budget)
  - `LimitRange/tenant-default-limits` (defaults + maxima)

Tenant onboarding already defines namespaces under:
`platform/gitops/tenants/<orgId>/projects/<projectId>/namespaces/<env>/`
so the operator UX is “set one label in the Namespace YAML”.

Defaults:
- If no profile is set, use a safe default profile (“baseline” / `medium`).

### 2) Expand quota coverage beyond CPU/memory

Quota profiles should cover at least:
- CPU/memory totals:
  - `requests.cpu`, `requests.memory`, `limits.memory`
- Workload/service counts:
  - `pods`, and (optional) `services`
- Storage:
  - `requests.storage`, `persistentvolumeclaims`
- Control-plane protection via object-count quotas (v1 candidates):
  - `count/configmaps`
  - `count/secrets`
  - `count/jobs.batch`
  - `count/cronjobs.batch`
  - (optional) `count/httproutes.gateway.networking.k8s.io` if we want a hard cap on route churn

Object-count quotas are explicitly about protecting etcd and the control plane from API object spam.

### 3) Rate limiting as a tenant budget (separate enforcement layer)

Some “quotas” are not Kubernetes `ResourceQuota` resources:
- tenant ingress budgets (RPS / concurrency) so one tenant under attack does not starve the shared ingress gateway
- egress gateway/proxy budgets (RPS / concurrency / bandwidth) so one tenant cannot exhaust shared egress

This idea doc treats these as part of the tenant “budget profile”, but the detailed enforcement design is tracked separately:
`docs/ideas/2026-01-16-infrastructure-rate-limiting-and-ddos-resilience.md`.

### 4) Implementation shape (keep GitOps + minimize bespoke controllers)

Primary candidate: extend the existing Kyverno generate baseline with mutually-exclusive rules per profile:
- Multiple `generate-tenant-quota-*` rules matching `darksite.cloud/quota-profile=<...>` and generating the same object name (`ResourceQuota/tenant-quota`) with different `spec.hard`.
- Multiple `generate-tenant-default-limits-*` rules matching the same profile label and generating `LimitRange/tenant-default-limits` accordingly.
- Keep `synchronize: true` and `generateExisting: true` so changes are drift-resistant and apply to existing namespaces.

Secondary candidate (if profile count grows): store the profile table in a ConfigMap and have Kyverno pull it via `context.apiCall`, with the namespace label selecting the key.

## What is already implemented (repo reality)

- Tenant baseline quotas/limits exist today and are generated into tenant namespaces:
  - `platform/gitops/components/shared/policy-kyverno/policies/clusterpolicy-tenant-baseline-generate.yaml`
  - Design: `docs/design/policy-engine-and-baseline-constraints.md` (Constraint C)
- Tenant onboarding is Git-shaped under `platform/gitops/tenants/**` and already carries identity labels required by admission policies:
  - `platform/gitops/tenants/README.md`
  - `docs/design/multitenancy-label-registry.md` (includes planned `darksite.cloud/quota-profile`)

## What is missing / required to make this real

- Define the v1 profile set (names + budget tables) and document sizing assumptions.
- Decide whether profiles are per-namespace (v1) or whether we need org/project-level “aggregate budgets” (future).
- Implement profile selection in the baseline generator (Kyverno rule split or ConfigMap-driven lookup) and add smoke coverage:
  - changing a namespace from `small` → `large` updates `ResourceQuota/tenant-quota` and `LimitRange/tenant-default-limits`
  - exceeding each profile is deterministically denied at admission
- Update tenant templates/docs so quota intent is visible during onboarding PRs.

## Risks / weaknesses

- Mis-sizing: too-tight profiles cause friction; too-loose profiles negate protection.
- Profile explosion: too many profiles become hard to reason about; prefer few coarse presets.
- Kyverno gotcha: multiple generate rules producing the same object must be mutually exclusive to avoid conflicts.
- Namespace-scoped quotas don’t express “org-wide” budgets if an org has many namespaces (future scaling issue).

## Alternatives considered

- Per-namespace `ResourceQuota` committed directly under `platform/gitops/tenants/**`:
  - simple and explicit, but higher per-tenant YAML volume and easier to diverge from baseline intent.
- A dedicated `TenantQuota` CRD + controller:
  - expressive (aggregate budgets, multi-namespace), but adds bespoke controller maintenance.
- Hierarchical namespaces (HNC) or virtual clusters:
  - can model aggregate quotas better, but are a bigger architectural step and out of scope for v1.

## Open questions

- Do we need separate default profiles for `dev` vs `prod` (`<env>` == overlayMode), or should onboarding policy (human process) decide budgets?
- Which object-count quotas matter most for control-plane protection in practice (ConfigMaps/Secrets/Jobs/HTTPRoutes)?
- Should we ever allow CPU limits for tenant workloads, or keep “requests-only” CPU (current posture)?
- How should quota changes be audited/evidenced (e.g., evidence entry per tenant budget increase)?

## Promotion criteria (to `docs/design/**`)

Promote once:
- A concrete label contract exists (at least `darksite.cloud/quota-profile`) and is recorded in the label registry.
- A v1 profile table is defined and reviewed (numbers + rationale).
- A working GitOps prototype exists that:
  - applies different quotas/limits based on the profile label,
  - updates quotas/limits on label change,
  - has deterministic smoke tests proving enforcement.
