### High

#### backup-and-recovery
- Backups currently land on an in-cluster PVC; move tier-0 backups to an off-cluster target and document durability + access controls (keep the PVC only as a cache if needed). (ids: `dk.ca.finding.v1:postgres-base:6f5be3d22c913405c8edf00e4894a892e8c0645b73bf739e9979cdb18351ab97`)

#### capacity-and-performance-hygiene
- Low-memory overlays request `512Mi` but base tuning keeps `shared_buffers=512MB`; patch tuning knobs for low-mem overlays to avoid OOM/startup instability. (ids: `dk.ca.finding.v1:postgres-base:a29c11a55dd1b358579b2f7614215291a633a1258065d709d2c60ff4acd3a083`)

#### data-protection-at-rest
- Encryption-at-rest is not evidenced for the StorageClasses backing data/WAL and backup PVCs; require/enforce encrypted classes (or document guarantees + key custody/rotation expectations). (ids: `dk.ca.finding.v1:postgres-base:8aacfe14ab6926fd157d1e7f2207e100d902ca48873eb429a749e995241d7650`)

#### general-security-pass
- Base CNPG cluster enables superuser access; disable by default and require explicit, tightly-controlled opt-in overlays (apps use least-privilege roles). (ids: `dk.ca.finding.v1:postgres-base:586afc05063b58317495d7ceffba20690cfc19f7d08ebd4bf40f2621f4fcd516`)

#### high-availability
- Dev low-memory overlays set Postgres clusters to `instances=1`; document as dev-only and ensure market-ready deployments stay `>=3` with tested disruption behavior (node drain/rollout). (ids: `dk.ca.finding.v1:postgres-base:38a05f51902a8523caed7adc477f6ce38146ef474136b8c260002a3c803cd130`, `dk.ca.finding.v1:postgres-base:bc8a879e95feae7bc3974b76112854c540e4025ec487b39352a13a8795adfb77`)

#### non-root-containers
- Backup CronJob/warmup Job lack least-privilege `securityContext` defaults; set non-root + hardening (no privilege escalation, drop caps, seccomp, RO rootfs where feasible) and enforce via policy where applicable. (ids: `dk.ca.finding.v1:postgres-base:1521709e23b904e0f1abb3f2bb087f3d927b833e5917175fe5ecd984c8a28a1b`, `dk.ca.finding.v1:postgres-base:cf88a7e7aa8f48d1414e42fbad558bf39be9871b1af55b09274ac9f912261944`, `dk.ca.finding.v1:postgres-base:a2f6834273ae6fc2b0362e76e0b3704660eccf9d12c3964d311e1f2f4d04b880`, `dk.ca.finding.v1:postgres-base:af05c77346799e9f11a1b0f8e13f2b893c363b5438ab32fe9ea03e60b625f00b`)

### Medium

#### backup-and-recovery
- Backup job uses superuser creds (via env vars); switch to a dedicated least-privilege backup role/CNPG-native backups and tighten secret consumption (e.g., file-based `PGPASSFILE`). (ids: `dk.ca.finding.v1:postgres-base:291eb1b3f537126715a8cfe1eeb0abd349f0b789b63082f799c461e80ae3c46f`, `dk.ca.finding.v1:postgres-base:0a28c2ea3e37c039d8eb32b8f14695e4f77003200f4c5b7c239faae3e5dc4b0c`)

#### capacity-and-performance-hygiene
- Resource requests/limits are incomplete for the CNPG cluster and backup workloads; define bounds to avoid noisy-neighbor/scheduling surprises. (ids: `dk.ca.finding.v1:postgres-base:e0d46608e179f4dbd911b2b5907c4774b3b0182ab7cf03cda3519696ff2caea4`, `dk.ca.finding.v1:postgres-base:5f8826bb024fde1d6ead7c950cea9374096e1733eac02903cd365b0a275f3c2a`)

#### secrets-handling

#### upgrade-and-rollback-strategy
