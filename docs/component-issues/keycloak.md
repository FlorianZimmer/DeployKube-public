## Resolved Items
- **2026-03-01 – Removed temporary realm identities from defaults:** `deploykube-admin` no longer ships `temp-*` roles/groups; canonical `dk-*` and `dk-bot-*` group contract is enforced in the base realm template.
- **2026-03-01 – Removed `dk-breakglass` from admin realm template:** `deploykube-admin` realm template now keeps canonical platform/bot groups without an online breakglass group, aligned with offline-only breakglass posture. Follow-up rollout completed on Proxmox after restoring `platform-apps-controller` fan-out and fixing controller image compatibility; Proxmox and OrbStack runtime validation both confirm rendered realm templates no longer include `dk-breakglass`.
- **2026-03-01 – Bootstrap hook no longer hangs indefinitely on OrbStack:** hardened `platform-keycloak-bootstrap` script to time-box `kcadm` paths (including best-effort role/group grants and user creation), reduced bootstrap Job deadline to 15 minutes, and capped `mac-orbstack` Keycloak HPA max replicas to 2 to avoid dev cluster churn. Revalidated OrbStack + Proxmox end-to-end (realm discovery and Vault JWT RBAC behavior).
- **2026-02-23 – IAM mode E2E split formalized (CI quick/full + in-cluster smokes):** added live matrix runner `tests/scripts/e2e-iam-modes-matrix.sh` and workflow `.github/workflows/keycloak-iam-mode-e2e.yml` (`quick` on PR internal branches, `full` nightly/manual on self-hosted). Existing CronJobs remain the continuous in-cluster assurance signal; CI now adds targeted mode-matrix regression coverage.
- **2026-03-15 – IAM sync helper ownership restored:** `platform-keycloak-iam-sync` now installs `ConfigMap/keycloak/istio-native-exit-script` itself again, matching the mounted volume contract for `keycloak-iam-sync` and `keycloak-ldap-sync`. This unblocked stuck CronJob Pods after the Postgres cutover and restored clean Istio-native Job termination for the sync paths.
- **2026-02-22 – IAM completeness gaps closed:** implemented OIDC/SAML upstream group mapper reconciliation, LDAP sync CronJob path (`keycloak-ldap-sync`), hybrid redirect toggling across `primaryRealm` + `secondaryRealms`, DeploymentConfig-driven Keycloak upstream egress allowlist controller (`NetworkPolicy/keycloak-upstream-egress-managed`), and IAM handover baseline + helper script (`shared/scripts/keycloak-iam-handover.sh`).
- **2026-01-29 – Ingress-adjacent hostname overlays renderer retired:** `HTTPRoute/keycloak` hostnames and `Certificate/keycloak-tls` `dnsNames` are now reconciled by the tenant provisioner controller from the deployment config snapshot ConfigMap; the repo-side renderer/validator were deleted.
- **Keycloak HPA (2026-01-19):** added an `autoscaling/v2` `HorizontalPodAutoscaler` targeting the Keycloak CR scale subresource to keep a baseline of 2 instances (min=2/max=4, CPU `averageValue: 200m`) while scaling up on demand. Updated the Keycloak PDB to `maxUnavailable: 1` and configured Argo to ignore `/spec/instances` drift so it does not fight HPA.
- **Keycloak smoke Job (2026-01-07):** implemented `keycloak-smoke` Argo PostSync hook Job to verify realm endpoints and OIDC discovery.
- **Over-requested CPU/memory (2026-01-04):** Keycloak instances were configured with `requests cpu=1, memory=3Gi` and `limits cpu=2, memory=6Gi` which heavily over-reserved cluster capacity (observed steady-state ~`3m` CPU and ~`0.55Gi` working set per pod). Reduced to `requests cpu=250m, memory=1Gi` and `limits cpu=1, memory=2Gi` to restore schedulability without impacting functionality.
- **Bootstrap hook flakiness from Istio proxy warm-up (2025-12-29):** `Job/keycloak-bootstrap` ran with an injected sidecar but without `holdApplicationUntilProxyStarts`, so early HTTP calls could hit `connection refused` (Envoy not ready yet). This caused the PostSync hook to fail and left required realms/clients unprovisioned (downstream impact: `forgejo-team-sync` couldn’t obtain tokens). Fix: add `proxy.istio.io/config: {"holdApplicationUntilProxyStarts": true}` to the bootstrap Job pod template so the script only starts once Envoy is ready.
- **Token helpers failing with `invalid_grant: Account is not fully set up`**: `shared/scripts/vault-token.sh` / `shared/scripts/argocd-token.sh` use the Keycloak password grant for non-interactive automation. Keycloak can reject the grant when the automation user is missing required profile fields (notably `firstName`/`lastName`, and sometimes `email`), even when `requiredActions=[]`. Fix: `platform-keycloak-bootstrap` now forces `firstName`/`lastName`/`email` and clears `requiredActions` for the developer + automation users during reconciliation, so the password grant stays usable after clean/preserve bootstraps (Dec 22, 2025).
- **Grafana OIDC “Client not found”**: Grafana redirected to Keycloak with `client_id=grafana`, but the `deploykube-admin` realm templates did not define that client. Fix: add a `grafana` OIDC client to the `deploykube-admin` realm template and project the client secret into the `keycloak` namespace for bootstrap rendering (Dec 15, 2025).
- **Keycloak Postgres backups failing (missing `password` key)**: `keycloak-postgres-superuser` was rendered by External Secrets with only the templated `username` field; the fetched `password` value was dropped, so the backup CronJob could not populate `PGPASSWORD`. The ExternalSecret template now includes `password: "{{.password }}"` (Dec 15, 2025).
- **Keycloak Postgres backups failing (Istio `ISTIO_MUTUAL` vs non-mesh CNPG pods)**: Keycloak’s CNPG pods run without sidecars, but the backup Job initially sent `5432/TCP` via Envoy (mesh default `DestinationRule *.local: ISTIO_MUTUAL`), yielding `connection refused`. The backup pod now excludes outbound port `5432` from sidecar interception (Dec 15, 2025).
