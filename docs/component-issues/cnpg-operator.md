## Resolved Issues

- **2026-03-15 – CNPG RBAC finding reframed as a documented internal control-plane exception:** the platform Postgres API now fronts all current CNPG consumers and tenant-facing CNPG usage is blocked, so the remaining upstream CNPG ClusterRole breadth is no longer tracked as an active hardening backlog item. It remains a suppressed exception, not a least-privilege fix: CNPG still retains broad internal control-plane RBAC and must keep its compensating controls (`tenant-deny-cnpg`, tenant AppProject/RBAC deny, `cnpg-system` isolation, HA, digest pinning) in place until DeployKube replaces or further decomposes that backend privilege model.
- **2026-03-15 – platform-owned Postgres API now fronts all current CNPG consumers:** Harbor and the disposable IDLab PoC have now joined Keycloak, Forgejo, and PowerDNS on `data.darksite.cloud/v1alpha1 PostgresInstance`, and `platform-postgres-controller` now supports the class-driven disposable/no-backup posture needed to retire the last raw CNPG consumer manifests without changing live service contracts.
- **2026-03-15 – platform-owned Postgres API first slice implemented:** DeployKube now ships `data.darksite.cloud/v1alpha1` with `PostgresClass` and `PostgresInstance`, a dedicated `platform-postgres-controller`, and Keycloak as the first internal consumer. CNPG remains the backend, but raw `postgresql.cnpg.io` is no longer the only internal contract.
- **2026-03-15 – CNPG `cnpg-system` traffic policy implemented:** the component now ships a default-deny network policy set plus explicit DNS/kube-apiserver egress and kube-apiserver/webhook ingress exceptions for `cnpg-system`; the README now documents the policy set and validation checklist, and Proxmox validation proved the operator stayed healthy and CNPG server-side dry-run still reached admission successfully. Resolved findings `dk.ca.finding.v1:cnpg-operator:b509d87fb8c56c8c1191b961125ad952db5b92cbc04354410bd941d00c77b0ea` and `dk.ca.finding.v1:cnpg-operator:ef3507daa8f8cb69c2ab6bca5c2fa0aa4b4fa9dcc9478dd0d7d1699c62a5e78f`.
- **2026-03-15 – CNPG status-port egress restored after `cnpg-system` policy regression:** the initial default-deny policy missed operator egress to CNPG instance-manager status endpoints, which surfaced as `Instance Status Extraction Error: HTTP communication issue` on single-instance CNPG clusters (`dns-system`, `forgejo`, `keycloak`, `idlab`). The policy now explicitly allows `8000/TCP` egress to pods labeled `cnpg.io/podRole=instance`, restoring operator readiness extraction without reopening broad east-west access.
- **2026-03-15 – CNPG operator moved to tier-1 HA posture:** the deployable path now patches the operator to `replicas: 2`, keeps leader election, adds preferred hostname anti-affinity plus hostname topology spread, and ships `PodDisruptionBudget/cnpg-operator-cloudnative-pg` with `minAvailable: 1`; the HA tier labels were updated to `darksite.cloud/ha-tier=tier-1`, and the README now documents expected drain behavior. Resolved findings `dk.ca.finding.v1:cnpg-operator:6273558fad0605e938930e2a09e7d0dee0efac28f6869d27fef357cddc305635`, `dk.ca.finding.v1:cnpg-operator:b6f01b502d9a9322a13513f09e147c2e1d354104a9f30724a5244367215614a7`, `dk.ca.finding.v1:cnpg-operator:71dd5b1ed23434038c9af0e26929328b38141427196f9012bb9b5deb3409d267`, and `dk.ca.finding.v1:cnpg-operator:13bf7dc6c3e66dd35022d84ddfe4b4a1f84097586ef29af7c6330b4227667314`.
- **2026-03-15 – CNPG operator digest-pinned with explicit scan control:** the deployable CNPG operator image is now digest-pinned in the component `kustomization.yaml`, `OPERATOR_IMAGE_NAME` is patched to the same reviewed ref, the runtime artifact catalog carries the same pinned image, and the README now points to centralized Trivy/runtime-artifact validation as the explicit CI scan control. Resolved findings `dk.ca.finding.v1:cnpg-operator:60c6fe9c074ef9c2ce9dadb2fd16e327eaa861e1ddf6d1f640740b40cb135f85`, `dk.ca.finding.v1:cnpg-operator:bbffcbe133f4256b1957f19818a86a028ac1333b863b7d8d08d15893d1c964e0`, and `dk.ca.finding.v1:cnpg-operator:e0a100882a9cfaa6557d936c010e7185bd6da872593fa4d1a67d084d5147f480`.

- **2026-03-14 – CNPG docs and tenant guardrails aligned to runtime truth:** documented the full CRD surface, Argo-level SSA path, chart provenance verification flow, webhook cert lifecycle, rollback checklist, fixed sizing/VPA posture, and explicit third-party API exception in the component README + shared data-services design note; added `tenant-deny-cnpg` plus smoke-suite coverage so tenant namespaces are admission-blocked from creating CNPG resources directly. - **2026-01-09 – CNPG operator namespace hardened**: `cnpg-system` is intentionally out-of-mesh (`istio-injection=disabled`), so it can enforce `pod-security.kubernetes.io/enforce=restricted` without requiring Istio init container privileges.
- **2025-12-29 – Operator CrashLoop due to guardrails blocking webhook updates:** `cnpg-operator-cloudnative-pg` exits on startup when it cannot `UPDATE` `cnpg-mutating-webhook-configuration` / `cnpg-validating-webhook-configuration` (PKI/webhook wiring). Fix: allowlist the CNPG operator service account for `UPDATE` on `cnpg-*` webhook configurations in `deploykube-access-guardrails-self-protection`.
- **2025-12-29 – Postgres clusters stuck due to RBAC guardrail:** CNPG clusters could not create their per-cluster `Role`/`RoleBinding` auxiliary objects (e.g. `Role/postgres`) because `deploykube-rbac-mutation-lock` denied those mutations. Fix: allowlist the CNPG operator service account for `CREATE/UPDATE` on `roles`/`rolebindings` labelled `cnpg.io/cluster`.
