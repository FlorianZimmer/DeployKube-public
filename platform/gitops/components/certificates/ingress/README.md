# Platform Ingress Certificates

Managed collection of `Certificate` resources that provision TLS credentials for public platform endpoints via cert-manager.

Mode behavior is controlled by `DeploymentConfig.spec.certificates.platformIngress.mode`:
- `subCa|acme`: controller reconciles per-host `Certificate` resources.
- `wildcard`: controller projects one wildcard TLS `Secret` from Vault/ESO and Gateway listeners reference that single secret.

Tier S tenant workloads wildcard certificates are **controller-owned** (tenant provisioner) and are not part of this component:
- `Certificate/istio-system/tenant-<orgId>-workloads-wildcard-tls` (`dnsNames`: `*.<orgId>.workloads.<baseDomain>`)

## Reconciliation (controller-owned)

This component is **controller-owned** and reconciled by the tenant provisioner controller from the DeploymentConfig snapshot ConfigMap (`argocd/deploykube-deployment-config`).
There is no repo-side “render then commit” overlay workflow.
Runtime truth for platform endpoint `Certificate` resources lives in `tools/tenant-provisioner/internal/controllers/platform_ingress_certificates_controller.go`.

## Subfolders

| Path | Purpose |
| ---- | ------- |
| `.` | Component root (intentionally empty; GitOps does not apply manifests here). |
| `base/` | Contract-only placeholders (non-deployable; hostnames are `*.placeholder.invalid`). |
| `overlays/<deploymentId>/` | Legacy (retired): kept as empty dirs only to satisfy repo overlay layout contracts. |

## Container Images / Artefacts

- None directly. Relies on the `cert-manager` controller image to process the resources.

## Dependencies

- **Components:** `certificates/cert-manager` must be running and healthy.
- **ClusterIssuer:** `step-ca` must be present and valid.
- **Namespace:** `istio-system` must exist (managed by Istio).

## Communications With Other Services

### Kubernetes Service → Service calls
- **cert-manager:** Watches these `Certificate` resources.
- **Istio Gateway:** Consumes the resulting Secrets.

### External dependencies (Vault, Keycloak, PowerDNS)
- **Step CA:** Signs the CSRs generated by these manifests.

### Mesh-level concerns (DestinationRules, mTLS exceptions)
- None. These are data-at-rest (Secrets) used by the ingress gateway.

## Initialization / Hydration

- Applied as part of the `platform-apps` root bundle.
- There is no specific initialization job; `cert-manager`'s control loop handles the provisioning.

## Argo CD / Sync Order

- **Sync Wave:** Default (0/unset).
- **Dependencies:** Implicitly depends on `cert-manager` being available. If `cert-manager` is down, these resources will stay in `False` readiness but won't block the sync state (unless health checks enforce it).

## Operations (Toils, Runbooks)

- **Sync:** `argocd app sync certificates-platform-ingress`.
- **Verify:**
  ```bash
  kubectl -n istio-system get certificate
  # Should show READY=True
  kubectl -n istio-system get secret <name>
  # Should contain tls.crt and tls.key
  ```
- **Renewals:** Handled automatically by cert-manager.

## Customisation Knobs

- **New Hosts:** extend `platform/gitops/deployments/<deploymentId>/config.yaml` (`spec.dns.hostnames.*`), then let the controller reconcile.
- **Tenant wildcard certs:** author a `tenancy.darksite.cloud/v1alpha1 Tenant` object; the tenant provisioner controller reconciles the wildcard `Certificate` (no overlay render step).
- **DNS definitions:** derived from DeploymentConfig; avoid hand-editing overlay YAML.
 
Repo-only cutover guardrail:
`./tests/scripts/validate-certificates-ingress-controller-cutover.sh`

## Oddities / Quirks

- **Namespace:** These live in `istio-system` because the Istio Gateway (which is not in this component) needs to mount them, and cross-namespace Secret mounting is not standard in Istio without extra config.

## TLS, Access & Credentials

- **Secrets:** Generates standard TLS secrets (`kubernetes.io/tls`).
- **Access:** Only the Ingress Gateway pod needs read access to these secrets.

## Dev → Prod

- **Promotion:**
  - Keep `spec.dns.hostnames.*` in the DeploymentConfig as the single source of truth.
  - Keep `spec.certificates.platformIngress.mode` in the DeploymentConfig as the single source of truth for issuer selection.
  - Let the tenant provisioner reconcile the per-host `Certificate` objects; the retired `overlays/<deploymentId>/` directories are historical scaffolding only.

## Smoke Jobs / Test Coverage

- **Automated:** `platform/gitops/components/certificates/smoke-tests/` includes `CronJob/cert-smoke-ingress-readiness` (all ingress `Certificate` resources reach `Ready=True`) and `CronJob/cert-smoke-gateway-sni` (Gateway presents the expected SNI certificate).
- **Manual:** Check status via `kubectl -n istio-system get certificate`. All must be `Ready`.
- **e2e:** The ultimate test is `curl https://<service>` ensuring valid TLS without warnings.

## HA Posture

- **Data Availability:** Rely on Kubernetes etcd (high availability) to store the resulting Secrets.
- **Controller Availability:** depends on `cert-manager` availability.
- **Failover:** If `cert-manager` is down, existing Secrets remain valid until expiry. Renewals will fail.

## Security

- **Access:** Secrets are stored in `istio-system`, restricted to usage by the Istio Ingress Gateway.
- **Rotation:** Managed automatically by `cert-manager` (short-lived internal certificates are recommended).
- **Zero Trust:** These certificates identify the "Front Door" of the platform.

## Backup and Restore

- **Configuration:** Restorable from Git (`platform/gitops/components/certificates/ingress`).
- **State:** If Secrets are lost, `cert-manager` immediately detects the mismatch and re-issues them (assuming `cert-manager` and `step-ca` are healthy).
- **Velero:** Should backup the `Certificate` resources to preserve status/history, but recreation from Git is safe.
