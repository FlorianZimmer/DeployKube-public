# platform/forgejo/jobs

One-shot Jobs that orchestrate Forgejo transitions.

1. `forgejo-cache-switch` – updates queue/cache/session to Valkey + restarts the Deployment (implemented).
2. `forgejo-admin-vault-sync` – writes the live Forgejo admin credentials into Vault (`secret/forgejo/admin`) once Forgejo is up, keeping ESO consumers (team bootstrap, Argo repo auth) aligned with the in-cluster secret instead of the early bootstrap placeholder.
3. `forgejo-db-seed` – scales the Forgejo deployment down, runs `forgejo dump --database postgres` inside the running pod to materialise a Postgres-friendly SQL dump on the shared PVC, drops/creates the `public` schema in CNPG using the superuser secret, imports the SQL as the application user, and then scales the deployment back up before writing the sentinel ConfigMap. The job requires access to `gitea-shared-storage` and both Postgres secrets.
4. `forgejo-db-switch` – repoints `forgejo-inline-config` to the Postgres DSN and restarts the Deployment after `forgejo-db-seed` runs. The script now blocks until the CNPG app secret (`forgejo-postgres-app`) exposes both `username` and `password` and the `postgres-rw` Service exists, preventing the Job from burning through its backoff window while the database stack is still converging.
5. `forgejo-oidc-config` – runs the Forgejo CLI inside the official container image, waits for the Keycloak-issued client credentials (and CA bundle) to be projected, and then creates or updates the OAuth2 authentication source (`openidConnect`, autodiscovery URL, scopes, and group claim) so the UI exposes a “Sign in with Keycloak” button even after PVCs persist between bootstraps. The job mounts `forgejo-oidc-client` for the credentials and `forgejo-oidc-ca` so HTTPS calls against the Keycloak discovery endpoint trust the same CA that cert-manager uses to sign the Keycloak ingress certificate.

The cache/database jobs run the `registry.example.internal/deploykube/bootstrap-tools:1.4` utility image (bundled `kubectl`, `psql`, etc.), whereas the OIDC configurator uses the Forgejo image so it can call `gitea admin auth …` directly from the same runtime as the Deployment. All scripts are surfaced through ConfigMaps generated by this directory.

## Istio + Argo hook gotchas (why Jobs “never finish”)

These Jobs are Argo CD PostSync hooks. Argo adds the `argocd.argoproj.io/hook-finalizer` finalizer, so a hook Job can be **impossible to delete** manually (even with `--force`) until Argo removes the finalizer.

If the Job runs in an Istio-injected namespace and the Envoy sidecar stays alive, Kubernetes keeps the Pod in `Running`, so the Job never reaches `Succeeded/Failed`, so Argo never removes the finalizer. This looks like: “Job is done, but it’s stuck forever”.

### Root causes we hit (Proxmox)
- **Early-exit paths skipped the Istio quit trap:** `forgejo-db-seed` exited early when the sentinel ConfigMap existed, but the script only installed the `trap deploykube_istio_quit_sidecar EXIT` later. Result: the main container exited `0`, but `istio-proxy` kept running, wedging the Job.
- **Wrong trust anchor published for Keycloak:** Keycloak’s TLS certificate was signed by a different CA than the one projected into `Secret/forgejo-oidc-ca`, so `gitea admin auth add-oauth` failed with `x509: certificate signed by unknown authority`. Fix: `keycloak-oidc-ca-sync` mirrors the cert-manager signing CA (`cert-manager/step-ca-root-ca`) into Vault `secret/keycloak/oidc-ca`, and ESO fans that into `forgejo-oidc-ca`.

### Preventing recurrence
- Scripts that run in Istio-injected Pods must install the Istio quit `trap` **before** any sentinel/validation logic that may `exit`.
- Hook Jobs use `argocd.argoproj.io/hook-delete-policy: …,BeforeHookCreation` so a resync can delete/recreate a previously failed/stuck hook without manual `kubectl delete` (and without fighting the Argo hook finalizer).
- The Proxmox overlay is now a thin Kustomize overlay that patches only the `OIDC_DISCOVERY_URL`; it no longer duplicates scripts/YAML, avoiding “fix landed in base but not in overlay” regressions.

### Emergency unstick (if you ever see this again)
1. Identify the hook Job + stuck Pod:
   - `kubectl -n forgejo get jobs,pods | rg forgejo-(db-seed|oidc-config)`
   - Symptom: main container `Completed`, `istio-proxy` still `Running`, Pod `NotReady 1/2`
2. Tell Envoy to exit so the Pod/Job can complete:
   - `kubectl -n forgejo exec pod/<pod> -c istio-proxy -- curl -fsS -XPOST http://127.0.0.1:15020/quitquitquit`
3. If Argo is still holding the finalizer, remove it (last resort):
   - `kubectl -n forgejo patch job/<job> --type=json -p='[{"op":"remove","path":"/metadata/finalizers"}]'`
