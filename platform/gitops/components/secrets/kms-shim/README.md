# KMS shim (seal provider)

This component provides a **minimal, stateless** “seal provider” service that provides the auto-unseal root-of-trust for OpenBao.

In v1, the shim emulates a small subset of Vault’s **transit** API surface (`/v1/transit/encrypt/*`, `/v1/transit/decrypt/*`) so OpenBao can use `seal "transit"` pointing at the shim, without running a second Vault/OpenBao cluster.

Design: `docs/design/openbao-secret-plane-kms-shim.md`

## What it is (and is not)

- ✅ A narrow envelope-encryption service used only for OpenBao auto-unseal.
- ❌ Not tenant-facing KMS (tenant “cloud-like KMS” should be OpenBao transit engine later).
- ❌ Not hardware-backed in v1 (PKCS#11-backed backend is tracked as future work).

## Runtime shape

- Namespace: `vault-seal-system`
- Workload: `Deployment/kms-shim` (2 replicas)
- Service: `kms-shim:8200`
- No PVCs.
- Tests: PostSync `Job/kms-shim-smoke` (encrypt/decrypt round-trip).

## Bootstrap inputs

This component expects two Kubernetes Secrets to exist (delivered via DSB + `secrets-bootstrap` in v1):

- `Secret/vault-seal-system/kms-shim-key`
  - key: `age.key` (Age identity file; the secret key line(s) generated by `age-keygen`)
- `Secret/vault-seal-system/kms-shim-token`
  - key: `token` (pre-shared token; passed as `X-Vault-Token`)

## Security posture

- Pods run with `readOnlyRootFilesystem`, `runAsNonRoot`, and drop all Linux capabilities.
- NetworkPolicy denies all egress and only allows ingress from:
  - `vault-system` (OpenBao client), and
  - same-namespace pods (smoke Job).
